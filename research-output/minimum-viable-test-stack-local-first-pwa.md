## Core recommendation

For a five-tester pre-field phase, the minimum viable stack is not "balanced" across layers. It should be intentionally lopsided. Put most of your rigor into pure deterministic domain tests in Mozilla-style Node/jsdom execution with GitHub-hosted libraries, a smaller but real slice into Dexie persistence tests with `fake-indexeddb`, and only a thin Chromium Playwright smoke layer for browser-only truths: service worker lifecycle, warm offline behavior, multi-page upgrade blocking, and a couple of "saved data survives reload" journeys. Keep React Testing Library narrow and functional, not exhaustive. Add Axe checks only on visible, high-value routes and visible modal/dialog states, and keep manual accessibility review for what automation cannot prove. That is the smallest stack that still yields believable confidence for a local-first PWA before live users touch it.

The right depth is asymmetric because the highest-confidence bugs in your app are not evenly distributed. Deterministic rule bugs are cheap to catch and expensive to miss, so they deserve heavy unit coverage. IndexedDB correctness is partly simulatable, so it deserves focused integration tests. Service worker, quota, installability, upgrade timing, and browser storage edge cases are environment problems, so they are poorly served by bigger Node-based suites and need only a few browser smokes plus a short manual device pass. Public guidance and repo patterns line up with that view: comparable projects emphasize behavior over implementation details, avoid testing third-party code, and keep browser testing targeted where the platform itself is the risk.

A good starting budget is roughly this: most tests in pure Vitest domain files, a small persistence suite using `fake-indexeddb`, a handful of RTL tests for task-critical UI wiring, five to eight Playwright Chromium smokes, and three to five Axe scans on visible routes or dialogs. No browser matrix in CI. No visual regression. No coverage gates that tempt people to write tests for lines instead of risks. No service-worker unit-test labyrinth.

## What comparable local-first stacks actually do

Excalidraw's public testing guide is revealing because it is practical rather than ceremonial. It documents a stack centered on Vitest, React Testing Library, `@testing-library/jest-dom`, `vitest-canvas-mock`, and `fake-indexeddb`. Its all-tests command runs type-checking, linting, formatting checks, and app tests together, and its published coverage thresholds are moderate rather than maximal. More importantly, its testing guidance explicitly says to test behavior rather than implementation details, avoid brittle structure assertions, avoid testing third-party code, and avoid over-mocking. That is exactly the posture a local-first PWA should copy before field validation: prove your logic and your user-visible behavior, but do not build a museum of fragile assertions around framework internals.

TinyBase shows a similar layering instinct from a different angle. Its package scripts split testing into unit, end-to-end, performance, docs, and production verification rather than pretending one enormous harness should do everything. That public shape matters more than the exact tool choices: it reflects a discipline of matching test type to risk type, which is the right move for local-first software where deterministic data semantics and platform behavior are different classes of risk.

RxDB is especially useful because it sits close to your Dexie/IndexedDB problem. Its docs explicitly recommend `fake-indexeddb` when running IndexedDB or Dexie-based storage in Node for unit tests, but its own repository still runs real browser test targets for storage backends, including Dexie, via Karma. In other words, the public lesson from RxDB is not "polyfill IndexedDB and call it done." The lesson is "polyfill for fast storage tests, then still run a real browser layer for storage-adjacent truths." That is the cleanest comparable pattern available for your stack.

PouchDB remains useful as a catalogue of browser-only IndexedDB pain. Its docs call out same-origin restrictions, private browsing failures, Firefox iframe/origin oddities, and Safari/WebView quota behavior. Whether or not you use PouchDB, those examples underscore the same architectural point: there is a class of storage bugs that does not exist in Node polyfills and only appears in real browsers under real policies. That class is exactly why your Playwright layer should exist at all.

Taken together, those public artifacts point to a stable pattern. Mature local-first projects spend their effort on logic, storage, and a thin slice of real-browser validation. They do not win confidence by asserting every component prop permutation, snapshotting every DOM subtree, or expanding CI matrices before the product's own behavior has stabilized.

## Where fake-indexeddb stops being truthful

`fake-indexeddb` is good enough to justify a serious persistence-integration layer, but not good enough to be mistaken for a browser. Its own README is unusually candid: it is an in-memory implementation whose main use is testing IndexedDB-dependent code in Node, data is not persisted to disk, and its latest published IndexedDB Web Platform Test pass rate trails real browsers. The published comparison puts `fake-indexeddb` below Chrome, Firefox, and Safari, and specifically notes that worker and cross-origin-isolation cases are omitted from the fair comparison because they are not relevant to Node. That is a green light for CRUD, indexing, and transaction-level tests, and a red light for browser-lifecycle claims.

The trustworthy uses of `fake-indexeddb` are the ones that stay inside deterministic storage semantics. Use it to verify schema declarations, uniqueness constraints, query behavior, Dexie transaction rollback on error, idempotent writes, and migrations that occur within a single process opening a newer schema. Dexie and RxDB both support that shape directly: Dexie can operate atop a swapped IndexedDB implementation, and RxDB's Dexie storage docs explicitly recommend `fake-indexeddb` for unit tests in Node.

The classes of bugs that escape `fake-indexeddb` are the ones caused by browser policy, lifecycle, or inter-client coordination. Real browsers require upgrade acknowledgements from other open tabs before schema changes proceed, and Dexie exposes that through `blocked` and `versionchange` handling. The IndexedDB docs also warn that browser shutdown aborts transactions and closes database connections unexpectedly, which is exactly the kind of condition your Node polyfill suite does not reproduce in a meaningful way. Those are Playwright or manual-device problems, not reasons to add more Vitest code.

Quota and eviction are another hard boundary. `fake-indexeddb` has an open issue asking for a way to simulate `QuotaExceededError`, precisely because reproducing true storage exhaustion is hard even in an actual browser. Meanwhile, current MDN guidance says IndexedDB, Cache, and OPFS writes can fail with `QuotaExceededError`, that private browsing often uses different quotas and clears data at session end, and that Safari may proactively evict data for origins not used regularly. If local persistence is mission-critical, no amount of fake-indexeddb work will substitute for at least one real-browser or real-device rehearsal of quota-adjacent behavior.

So the handoff is simple. Vitest plus `fake-indexeddb` owns storage logic. Playwright takes over at the moment your question involves more than one page, a real browser lifecycle, a service worker, quota policy, installability, browser shutdown, or origin policy. That is the clean boundary that keeps the fast suite honest instead of inflated.

## Service workers and accessibility without overbuilding

Service-worker testing is worth doing, but only in the narrow slice that browsers actually make reliable. Playwright's current service-worker guide says service-worker support is Chromium-only, exposes APIs to wait for activation, inspect service-worker-owned requests, and check whether a response came from a service worker, but also documents limitations such as inability to route updated service-worker main-script requests. The Vite PWA testing guide is consistent with that reality: it describes build testing with Vitest and in-browser testing with Playwright, specifically using Chromium and a started server. That means your CI should run service-worker smoke tests in Chromium against a built app, not against a Vite dev session and not across a broad browser matrix.

What is reliably worth automating in Playwright is modest but valuable. Test offline-after-first-load rather than offline-on-first-visit. The service worker lifecycle docs are blunt: by default, a page that loads without a service worker is not controlled by one until a later navigation or refresh, unless you intentionally use `clients.claim()`, and even then timing can be racey. So the right smoke is: first load online, wait until the worker is active and in control, save local data, go offline, reload, and verify the app shell and saved data still work. Also automate one update-path smoke if you expose a reload prompt: Workbox's update guidance centers on the waiting worker, a user prompt, `messageSkipWaiting()`, and observing the new worker take control. vite-plugin-pwa's React docs show this same shape through `useRegisterSW()` and a reload button that calls `updateServiceWorker(true)`.

Two more browser smokes are defensible if your team can afford them. One is a multi-page blocked-upgrade test, where page A keeps the old schema open and page B loads a new build, so you can verify your upgrade UX instead of discovering a deadlock in the field. The other is a "persisted Dexie data survives normal reloads and route changes" test, which is boring but high-value because it exercises the actual browser storage stack. These are exactly the kinds of conditions Dexie and IndexedDB document, and exactly the kinds of things fake-indexeddb cannot certify on its own.

What should stay out of CI for now is equally important. Do not try to automate full install-prompt flows as a gate; `beforeinstallprompt` is limited-availability and installation behavior varies by browser. Do not treat periodic background sync as a CI responsibility; MDN still marks it experimental, and its real-world behavior depends on install state and browser support. Do not spend time simulating OS backgrounding, iOS home-screen resumes, storage-pressure eviction, or "network process crashed while IndexedDB was open" bugs inside your normal suite. Those belong in a short manual device pass before the cohort opens, especially on Safari/iOS and Chrome/Android.

Accessibility should be disciplined the same way. Playwright's accessibility guide recommends using Axe tags `wcag2a`, `wcag2aa`, `wcag21a`, and `wcag21aa` to match the automated-check scope used by Accessibility Insights. Deque's tag docs confirm those tags and show that `best-practice` is separate from WCAG conformance tags, while `wcag22aa` is available if and when you expand scope. For your phase, block on WCAG 2.1 A/AA tags only. Run `best-practice` as advisory if you want it, but do not turn it into a pre-field merge gate.

Also keep the suppressions narrow. Playwright's own guide says the safe approaches are `exclude()` for known, tightly scoped problem areas, `disableRules()` only when one specific rule is noisy, or violation-fingerprint snapshots for known debt. Deque's API docs remind you that hidden regions are not tested at all, so dialogs, menus, and drawers must be opened before scanning. Storybook's accessibility docs also show one practical false-positive-adjacent case: the `region` rule is often disabled in isolated component stories because stories are not full pages. For app-level route tests, keep `region` on. For component-only stories, disabling it is reasonable. Color-contrast is mostly trustworthy, but Deque's own rule docs mark some outcomes as incomplete/manual review, and issue threads show complex pseudo-element or gradient backgrounds as a recurring edge case. The rule of thumb is simple: treat Axe failures as real until proven otherwise, but if the only suspect is a complex CSS painting trick, verify visually before you suppress.

## The blueprint

The blueprint below keeps Node fast, browser truth thin, and CI cheap. It is tuned to the documented behavior of Dexie, `fake-indexeddb`, Playwright service workers, vite-plugin-pwa's React registration flow, and Axe's tag system.

### Layer split

| Layer | Investment now | What it should prove | What it should not try to prove |
|---|---|---|---|
| Vitest domain tests | Heavy | All verdict logic, edge cases, invariants, serialization boundaries | Browser behavior, Dexie wiring, React rendering |
| Vitest + fake-indexeddb | Focused | Schema, indexes, transaction rollback, saved records, simple migrations | Quota, eviction, multi-tab blocking, install/update lifecycle |
| React Testing Library | Thin | Critical task wiring, visible error states, form-to-domain mapping | Every component variation, snapshot churn, CSS fidelity |
| Playwright Chromium | Thin but real | Warm offline behavior, retained local data, service-worker control, upgrade smoke | Full browser matrix, visual regression, device-cloud coverage |
| Axe in Playwright | Thin and strict | Obvious WCAG 2.1 A/AA violations on visible flows | Manual-only issues like meaning, task clarity, full keyboard/screen-reader UX |

### Representative domain test (reference)

```ts
import { describe, expect, it } from 'vitest';
import { decideVerdict, type RulesInput } from './decision-engine';

describe('decideVerdict', () => {
  it.each<[string, RulesInput, 'progress' | 'hold' | 'deload']>([
    [
      'progress when adherence is high, flags are zero, and stability is strong',
      { adherencePct: 0.92, stabilityDays: 18, redFlags: 0, severeSymptoms: false },
      'progress',
    ],
    [
      'hold when adherence is decent but stability is marginal',
      { adherencePct: 0.78, stabilityDays: 6, redFlags: 0, severeSymptoms: false },
      'hold',
    ],
    [
      'deload when severe symptoms are present regardless of other positives',
      { adherencePct: 0.95, stabilityDays: 25, redFlags: 0, severeSymptoms: true },
      'deload',
    ],
    [
      'deload when multiple red flags accumulate',
      { adherencePct: 0.85, stabilityDays: 14, redFlags: 3, severeSymptoms: false },
      'deload',
    ],
  ])('%s', (_name, input, expected) => {
    expect(decideVerdict(input)).toBe(expected);
  });

  it('is deterministic for identical inputs', () => {
    const input: RulesInput = {
      adherencePct: 0.81,
      stabilityDays: 10,
      redFlags: 1,
      severeSymptoms: false,
    };
    expect(decideVerdict(input)).toBe(decideVerdict(structuredClone(input)));
  });

  it('never improves the verdict when severeSymptoms flips from false to true', () => {
    const safer: RulesInput = {
      adherencePct: 0.88,
      stabilityDays: 15,
      redFlags: 0,
      severeSymptoms: false,
    };
    const riskier: RulesInput = { ...safer, severeSymptoms: true };
    expect(decideVerdict(safer)).not.toBe('deload');
    expect(decideVerdict(riskier)).toBe('deload');
  });
});
```

### Representative persistence integration test (reference)

```ts
import Dexie, { type EntityTable } from 'dexie';
import { describe, expect, it } from 'vitest';

type DecisionRecord = {
  id: string;
  verdict: 'progress' | 'hold' | 'deload';
  savedAt: string;
};
type AuditRecord = { id: string; decisionId: string; action: 'save' };

class AppDb extends Dexie {
  decisions!: EntityTable<DecisionRecord, 'id'>;
  audit!: EntityTable<AuditRecord, 'id'>;

  constructor(name = 'app-db-test') {
    super(name);
    this.version(1).stores({
      decisions: 'id, verdict, savedAt',
      audit: 'id, decisionId, action',
    });
  }
}

async function saveDecisionWithAudit(db: AppDb, decision: DecisionRecord, auditId: string) {
  await db.transaction('rw', db.decisions, db.audit, async () => {
    await db.audit.add({ id: auditId, decisionId: decision.id, action: 'save' });
    await db.decisions.put(decision);
  });
}

describe('AppDb transaction behavior', () => {
  it('rolls back the decision write when the audit insert fails', async () => {
    const db = new AppDb();
    await db.open();
    await db.audit.add({ id: 'duplicate-audit-id', decisionId: 'seed', action: 'save' });
    await expect(
      saveDecisionWithAudit(
        db,
        { id: 'decision-1', verdict: 'hold', savedAt: new Date().toISOString() },
        'duplicate-audit-id',
      ),
    ).rejects.toThrow();
    expect(await db.decisions.get('decision-1')).toBeUndefined();
    await db.close();
  });
});
```

### Warm-offline smoke (reference)

```ts
import { test, expect } from '@playwright/test';

async function waitForServiceWorkerControl(page: import('@playwright/test').Page) {
  await page.evaluate(async () => {
    const registration = await navigator.serviceWorker.getRegistration();
    if (registration?.active?.state === 'activated' && navigator.serviceWorker.controller) return;
    await new Promise<void>((resolve) => {
      navigator.serviceWorker.addEventListener('controllerchange', () => resolve(), { once: true });
    });
  });
}

test('works offline after first online load and preserves locally saved state', async ({
  context,
  page,
}) => {
  const swPromise = context.waitForEvent('serviceworker');
  await page.goto('/');
  await swPromise;
  await waitForServiceWorkerControl(page);

  await page.getByLabel('Subject').fill('athlete-42');
  await page.getByRole('radio', { name: 'Hold' }).check();
  await page.getByRole('button', { name: 'Save locally' }).click();
  await expect(page.getByText('Saved offline')).toBeVisible();

  await context.setOffline(true);
  await page.reload();

  await expect(page.getByText('Offline mode')).toBeVisible();
  await expect(page.getByLabel('Subject')).toHaveValue('athlete-42');
  await expect(page.getByRole('radio', { name: 'Hold' })).toBeChecked();
});
```

## Trust invariants map

- **The rules engine returns the right verdict for known and boundary cases** -> pure Vitest domain tests, table-driven and exhaustive around your decision table.
- **The rules engine is deterministic and does not improve a verdict when inputs worsen monotonically** -> pure Vitest domain tests first; a tiny fast-check layer only if invariants stay compact.
- **A logical save is atomic in local storage** -> Vitest + Dexie + `fake-indexeddb`, by forcing transaction failure and asserting rollback.
- **IndexedDB schema upgrades do not wedge when another tab is open** -> Playwright multi-page test (`blocked` / `versionchange`).
- **The app works offline after the first online load and preserves saved local data** -> Playwright Chromium smoke against a production build.
- **Your service-worker update prompt actually leads to the new version taking control** -> one Playwright Chromium update-flow smoke tied to waiting/skip-waiting/`controllerchange`.
- **Visible critical screens do not ship with obvious WCAG 2.1 A/AA defects** -> Playwright + Axe on visible routes and visible modal states only.
- **Safari/iOS storage policy, eviction, installability, and embedded WebKit weirdness will not destroy user trust** -> not CI; short manual device pass, then cohort reality.

## Deliberate under-investment

- No component-testing empire: test behavior, not implementation details.
- No coverage % as a release gate; collect for visibility only.
- No visual regression yet; UI is still moving.
- No cross-browser CI matrix; Chromium only plus one short manual Safari/iOS and Chrome/Android pass.
- No unit-testing Workbox/vite-plugin-pwa internals; treat SW as an integration boundary.
- No property-based testing expansion on day one beyond one or two crisp invariants.
- No pretending automated accessibility means "accessible."

## End state

Before a five-person cohort, the stack should prove:

- the rules engine,
- Dexie writes,
- offline-after-warm-load,
- update behavior once,
- obvious WCAG 2.1 A/AA violations on the routes people will touch.

Anything beyond that is optional maturity work. If the suite starts to look more polished than the product, it is already too big.
