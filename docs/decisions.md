| D91 | M001 go/no-go evidence threshold: 5+ testers each complete 2+ sessions within 14 days; kill signal if fewer than 3 of 5 start a second session within 14 days. Read in three layers: (1) kill-floor is a hard no-go screen; (2) go-bar is **permission to keep building and keep testing**, not proof of durable value; (3) the raw count is read in bands — `0-1/5` strong negative, `2/5` ambiguous, `3/5` weak pass of floor (`continue investigating`, not `go`), `4-5/5` the first genuinely encouraging outcome. A bare threshold pass must be upgraded by at least one enrichment signal (unprompted return, second session >48h after the first, or a third session / concrete scheduling commitment) **plus** clean instrumentation: classify every second-session start as `self-initiated` vs `human-prompted` (concierge/founder nudging can manufacture accountability the product does not yet create); capture adherence dimensions (active minutes, full-completion vs mid-flow abandon, full-flow vs narrow-slice engagement) and context logs for completed **and missed** sessions (location, surface, weather, time-of-day, company, blocker); after a tester crosses the 2-in-14 line, ask the Ellis / Superhuman "how disappointed would you be if you could no longer use this?" conviction check. `>50%` review completion is an interpretation aid, not an independent gating metric. **Amended by `D119`:** field-test artifact shifts from v0a to v0b; thresholds and enrichment requirements unchanged. | Behavioral retention (repeat sessions) is more reliable than stated interest or waitlists. The kill-floor (`3/5` return) is calibrated against adjacent small-cohort pilot evidence and the self-coached outdoor context. The go-bar is intentionally a sanity-check rather than a PMF threshold because (a) at n=5 the exact 95% binomial CI on `3/5` spans ~15% to ~95%, so a single ratio cannot locate the true retention rate, and (b) novelty-effect literature shows a single return visit inside two weeks can still be curiosity. Concierge / founder nudging is a specific contamination risk for self-coached products: human accountability can manufacture adherence the product itself does not yet create, so self-initiated vs prompted returns are tracked explicitly. ## Ruled out (for now)


| Item                                                 | Reason                                                                                                                                                                                                             |
| ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Social/community feed                                | Not core to training workflow; distraction risk                                                                                                                                                                    |
| Video analytics / computer vision                    | Too complex for MVP; manual logging is sufficient                                                                                                                                                                  |
| Open-ended AI chat as primary UX                     | Structured objects and workflows are the product stance                                                                                                                                                            |
| AI-generated training plans or progression logic     | High risk of hallucinated load plans, omissions, and automation bias; strictly deterministic generation only                                                                                                       |
| Indoor volleyball support                            | Beach-specific focus is a feature, not a limitation                                                                                                                                                                |
| Monetization in v1                                   | Validate the loop first; monetization design captured in research                                                                                                                                                  |
| Community challenges                                 | v1.5+ at earliest; need proven single-user loop first                                                                                                                                                              |
| CRDTs or real-time collaborative editing in Phase 1  | No validated need for concurrent editing; async coach review is sufficient and avoids premature complexity                                                                                                         |
| Server-authoritative data model                      | Conflicts with the local-first product principle; the device is the source of truth                                                                                                                                |
| ACWR-based risk scoring or "danger zone" messaging   | Evidence is contested (statistical artifacts, no causal proof, method-sensitive results); shipping it risks false authority and overconfident safety claims. Spike avoidance uses conservative heuristics instead. |
| Deep recovery analytics (HRV, wearable integrations) | Unclear added value for M001's first wedge; increases complexity without validated need                                                                                                                            |
| Return-to-play guidance for specific injuries        | Quickly becomes clinical/medical advice; should be clinician-led, not app-driven                                                                                                                                   |
| Centralized coach marketplace / platform-sourced coach network (Future / on-demand expert access model) | Requires platform-led coach recruiting, matching, QA, and utilization management — a software-enabled services business, not a lightweight extension. Public evidence (Future at $199/month) and U.S. BLS wage medians (~$46k/year for fitness trainers and coaches/scouts) make that line economically workable only when dedicated human coaching is the whole product. A niche self-coached sport with no existing coach network is structurally the wrong context for this model. `D106` freezes BYOC-lite as the shape instead. Revisitable only on strong post-clipboard pull evidence against the four `D108` metrics. See `docs/research/coach-facing-business-models.md`. |
| Direct-to-coach SaaS as a standalone motion (coach seat SKU before athlete-side value is proven)        | Coach buyer is fragmented and often self-employed (IBISWorld); winners in the coach-SaaS category prices gently, add payments/branding/discovery, or sell into organizations. A product without existing coach distribution should not assume direct-to-coach SaaS will rescue growth. `D107` reserves monetization to the athlete side until repeated weekly coach usage appears.
