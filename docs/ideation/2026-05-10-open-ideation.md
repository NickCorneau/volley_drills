### A1 — Script-derived founder-use ledger + auto-computed D130 gate state

**Description:** Convert the founder-use ledger from a hand-written markdown self-report into a script-derived count-of-record. Add `npm run ledger:derive` that reads Dexie session data, git log, and the markdown ledger to emit a canonical week-by-week count table with diffs against the prior week. Auto-compute D130 conditions (1, 2, 3) and early triggers (a–e) from session data into a `gate-state.json` artifact, and render a rolling 8-week panel in the diagnostics report. The markdown ledger stays the narrative; the script becomes the count-of-record. The rolling-window primitive survives into M002 and every subsequent milestone.

**Warrant (direct):** Grounding names "ledger under-counts vs claimed 2x/week" as named pain; D130 Condition 1 is fail-trending under standard read (solo 1/7, set 0/3); the 2026-05-21 final read depends on the ledger; D135 source-validity gating requires countable evidence to distinguish trigger hits from feature-wishes.

**Rationale:** The single biggest threat to D130 Condition 1 is ledger under-count from self-report drift. Derivation makes the gate harder to lie to, makes the 2026-05-21 final read a routine snapshot, and aligns with docs-first / source-validity-gated discipline by making the ledger a derived artifact. Rolling 8-week panel turns founder-use into a reusable posture across milestones.

**Downsides:** Requires founder to keep using in-app capture honestly (Dexie session records are the input). Risks brittleness if Dexie loses session records on iOS 7-day purge — must reconcile against markdown ledger so old records aren't silently lost. A telemetry pipe would harden against the purge but crosses the local-first identity and is a separate decision.

**Confidence:** 85%
**Complexity:** Medium

### A2 — Cross-session review card primitive + FSRS-backed scheduler (M002 substrate)

**Description:** Define the durable atomic artifact as a cross-session "review card" per skill (not per session). Sessions produce review cards as byproducts of Drill Check capture. A local FSRS-backed scheduler (MIT-licensed `fsrs-rs` compiled to WASM, hosted on the existing Dexie v6 stack) picks next-session candidates from falsification reads. Reframe practice units: a 90-second "moment" (block + cue + count + difficulty) is what M001 captures; a 7-day "arc" (one skill's confidence shift) is what M002 reads; the session is a container, not the atom. Courtside surfaces stay judgment-free; all interpretation moves to desk-time review (Photography RAW workflow).

**Warrant (external):** Chess opening trainers (Chessli, ChessAtlas, Anki+FSRS) operate this exact loop — "import-the-game, derive-the-review" — flagged in grounding as the strongest cross-domain analogy. FSRS 6.0 (May 2025, MIT-licensed) is WASM-ready. BeachVolleyballAI's "Other 6 Days" framing publicly names the same gap M002 is meant to close. Headspace minimal-progression literature supports the unit reframe.

**Rationale:** Today's Drill Check signal already contains the data needed to schedule next-session content; the system just doesn't use it. Closing the gap gives M002 a defensible substrate without claiming M002 scope. The review card gives the user something to *hold* on no-court days, addressing the recorded learned-preference worry that the app "feels like a data-entry form, not a main training tool." Exportability aligns with O20 (drill/catalog as durable external artifact).

**Downsides:** FSRS was designed for discrete-correctness domains (chess, language); volleyball Drill Check is noisier (RPE, partner mood, weather). Risk of overfitting to noise → founder silently overrides prescribed sessions, eroding trust. Needs deliberate input coarseness (streak gate + difficulty bucket only). M002-shaped; should pass through `ce-brainstorm` and `ce-plan` before catalog-coupled work lands.

**Confidence:** 75%
**Complexity:** High

### A3 — State-aware partner-readable cue registry with pictogram fallback

**Description:** Promote the 14-rule courtside copy contract from per-string lint to a `cueCadenceRegistry` that knows state — drill index, repeat count, partner-context, fatigue band, time-into-block. Each cue is authored once with cadence rules (fade-on-repeat, partner-read variant, time-into-block escalation); the 14 rules become validators on the registry. Add a fixed 3–5 phrase pair callout vocabulary per drill (Aviation CRM-style), printed on the courtside card. Add a pictogram-dense topo-style card (one glyph per slot, grade-style difficulty mark, equipment row, outcome glyph) so prose collapses below the fold. Stand up an "incident register" logging every confusion-class read-aloud failure mapped to the copy rule it falsifies.

**Warrant (direct):** 2026-05-10 founder+Seb session marked 3 drills `still_learning` with 0 streaks because partners reading copy aloud failed on "peak and flash / number drill" copy. First-time-runnability brainstorm is already in flight on this exact confusion class (2026-05-10).
**Warrant (external):** Aviation CRM canonical pair-coordination intervention; UIAA climbing topos as outdoor-readable instruction format; Apple Watch 2-second glance contract.

**Rationale:** The active in-flight brainstorm names this exact pain. The state-aware registry converts the 14-rule contract from per-string discipline into machine-enforced cadence; fade-on-repeat (O14) gets a registry implementation; partner-readability stops being a rule we write and becomes a slot in the data model. Pictograms survive sun + partner-reading distortion that prose does not. The incident register turns one bad session into a durable signal class rather than a patch.

**Downsides:** Collision risk with the active 2026-05-10 first-time-runnability brainstorm (`docs/brainstorms/`); needs coordination, not parallel authoring. **Confidence:** 80%
**Complexity:** Medium-High

### A4 — Visual-first cue discipline (15th courtside-copy rule + count-not-timer migration)

**Description:** Codify a falsifiable rule that any drill block longer than 30 seconds must lead with a visual cue (count, body posture, court spot, pictogram), with audio strictly as redundant backup. Add as Rule 15 to `.cursor/rules/courtside-copy.mdc`, peer to the existing 14 rules. Audit every existing block >30s and tag whether it satisfies the protocol; produce a count of failures. Move rep-natural blocks from time-bounded to rep-count-bounded; conditioning blocks (where time matters) become an explicit separate slot type. Each block-state screen has one designated audible primary cue that is informational without the screen.

**Warrant (direct):** Grounding documents the unresolved iOS PWA 30-second Web Audio lockscreen cutoff (still unfixed 2025–2026, Apple Developer Forums thread #762582); anti-pattern list warns against "broad audio rewrite without platform-boundary diagnosis"; recurring warmup/cooldown timing pain across V3/V4 ships, mostly when audio is primary signal.
**Warrant (external):** MIP > AMOLED in direct sun (Garmin product bifurcation); Apple Watch 2-second glance contract; golf rangefinder ONE canonical number per glance; kiteboarding "made to be invisible" aesthetic.

**Rationale:** The platform boundary is permanent for as long as iOS PWA constraints hold; treating it as a recurring bug rather than a design contract has cost three V3/V4 timing-fix passes. Promoting visual-first to a rule converts the cutoff from a bug-to-patch into a design parameter, and aligns courtside copy with how the watch generation already consumes single-glance information. Counts > timers cleanly handles pair-shareability (a count is partner-visible; a timer is not).

**Downsides:** Dropping the block timer affects existing pacing-dependent drills (warmup/cooldown — d25/d26/d28) with segment-paced design; needs careful migration so segment cues survive. Reverses some recent V3/V4 segment-timing investment. Founder + Seb might experience the change as regression before it lands as discipline.

**Confidence:** 85%
**Complexity:** Medium

**Confidence:** 80%
**Complexity:** Low

### A6 — Three foundational model-layer primitives: CaptureRow + Pair entity + ambient timestamps

**Description:** Extract three composable model-layer primitives. (a) **CaptureRow**: a tiny drill-agnostic Dexie record `(captureId, drillId, sessionId, kind, value, source, timestamp, sourceBackedRef?)` that routes all current per-drill capture (D133 difficulty, D134 streak gate) and every future capture kind through one shape. (b) **Pair entity**: a first-class `Pair (id, members[], created_at, last_session_at, shared_history_ref)` connecting Session→Pair; the current `session.partner_context` becomes a pair lookup. (c) **Ambient timestamps**: quietly stamp `drill_started`, `drill_completed`, `transition_seen`, `page_visible_resume`, `block_timer_fired` into IndexedDB on every event regardless of explicit logging. No new UI; pure model-layer foothold.

**Warrant (direct):** D132 pair-first re-read is referenced directly in D130 condition reads; `docs/prd-foundation.md` frontmatter has drifted (`last_updated` 2026-04-19 vs D132 2026-04-22+); O13 (persistent pair identity), O19 (pair differential), O17 (session-fit triage), O15 (self-estimate before count), O14 (cue cadence), and the founder-use-ledger 2026-05-09 entry on the 3+/4-player gap all reference data shapes that don't exist yet. Dexie v6 schema is the right add-only seam.

**Rationale:** Every future capture decision drops from "design schema + migrate" to "add a row kind." Every future feature that should be pair-aware (M002, weekly digest, asymmetric-skill nudge) becomes possible without retrofit. Ambient timestamps give every diagnostic, ledger, gate-state, and pair-asymmetry read a free dataset that grows weekly. Composes naturally with A1 (derived ledger reads timestamps), A2 (review cards read CaptureRow), and A3 (cue registry reads cadence stamps).

**Downsides:** Schema-touching even when additive carries migration risk; must include the Dexie `Cannot inject key into script value` defensive retry in the new write path. Ambient timestamps could expand storage faster than expected on iOS where 7-day purge is real — needs TTL or compaction strategy. CaptureRow as a generic envelope risks losing type-specific lint that current per-kind shapes give us; mitigate with discriminated unions on `kind`.

**Confidence:** 75%
**Complexity:** Medium-High

### A7 — Plan-shape snapshot test infrastructure

**Description:** Build a `planShape` snapshot harness that captures the structural shape of every generated session plan (slot archetypes, `sourceBackedReroutes` events, source-backed activations, cap consumption, single-chain compliance) and freezes it per representative input. Every generator change must explicitly opt-in to shape changes via snapshot update. Wire into `architecture:check` or as a peer `npm run plan-shape:check` CI gate.

**Warrant (direct):** `generatedPlanDiagnosticTriage.ts` is named unwieldy (53-entry triage, D140 policy packets); four dense decision passes (D141/D142/D143/D144) shipped against the schema in the last two weeks; 21 advisory screen→service imports under `architecture:check --strict`. Regression surface is real, growing, and currently policed by prose audits.

**Rationale:** Converts diagnostic triage from prose audits into code-enforced contracts. Second-order: every subsequent decision-pass costs less because the cost of being wrong about plan shape drops; M002 weekly loop boots with the same shape diffing on day one; future agent refactors of the decisions mega-table become safer; future Tier 1c authoring inherits the gate for free. Direct analog to how `focusCoverageAudit` codified the audit surface (180/180 covered, snapshot-tested).

**Downsides:** Snapshot tests can rot if updated reflexively without thought (Volleycraft already has `focusCoverageAudit` snapshot which has rebaselined cleanly — pattern is healthy). The harness must hit the right granularity — too coarse misses regressions; too fine churns. Initial authoring of representative inputs has real cost.

**Confidence:** 85%
**Complexity:** Medium

## Cross-pass rejection vision-flags

Three Pass B survivors are vision-misaligned in ways worth naming explicitly so they don't recirculate in a future open ideation pass.

### B1 — Screenless Audio Appliance — defer until platform changes

- **Conflict:** iOS PWA Web Audio cutoff at 30s on lockscreen is unfixed (Apple Developer Forums thread #762582, 2025–2026); a self-coached amateur who pockets the phone for >30s loses the cue. The 90% confidence number is much higher than the platform supports.

- **Disposition:** Defer until either iOS fixes the cutoff or the product moves to a native runtime. Not a current ship candidate.

### B5 — Mycelial Content Ingestion Engine — reject (P7 + D135 + cap discipline)

- **Conflict (P7):** "Agent-driven pipeline that extracts source drills … into an append-only Bucket B schema" puts GenAI in the critical content path. P7 ("Human owns the final plan; session generation is built on strict, deterministic rules without generative AI") and the strategic stance ("Generative AI is excluded from the critical path") both rule this out.
- **Conflict (D135):** D135 source-validity gating frames feature-wishes as INPUT, not trigger hits. An agentic ingestion bypasses the source-validity gate by definition.
- **Conflict (problem framing):** The captured BAB drill book is structurally complete (the 2026-05-04 BAB-complete-plan-builder ideation explicitly found this is comprehensive research, premature for operationalization). D101 gates 5 of 6 top-tier source-priority families for principled reasons. The catalog isn't the bottleneck; cap discipline is intentional.
- **Disposition:** Reject. The instinct toward "Pull from pre-mined archives" is already operationalized via the manual source-backed authoring path under D135.

### B6 — Sniper's Spotter (external-focus lock-in) — reject (P11 + collapsed warrant)

- **Conflict (P11):** "User must physically acknowledge this specific external focus point before the timer starts" is forced interrogation. P11 is rank-1 ("Recommend before you interrogate") and is the same anti-pattern that drove SetupScreen formification rework under D137.
- **Conflict (P12):** Adds a pre-block friction screen, which reads against "one clear action, one confidence signal, one reason to come back."
- **Conflict (external warrant):** Grounds itself in Wulf-tradition external-focus motor learning (g≈0.583), but Chua et al. 2024 Bayesian reanalysis collapsed that effect to g≈0.01–0.15. The intervention strength does not justify the friction cost.
- **Disposition:** Reject the **lock-in gate**. The **cue copy** ("Find the antenna" framing) is fine when used as a redundant secondary cue inside A3's cue registry — that's the right home for the surviving instinct.
