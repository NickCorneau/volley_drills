### P1 -- Manual fix required by 2026-07-21

- **[P1 / manual] Time-bomb on hard-coded `expiry: 2026-07-20` in cap-status test fixtures.** Reviewer: testing.
  - File: `scripts/test-validate-agent-docs.sh` (every cap-status fixture except `test_cap_status_passes_when_backlog_absent` and `test_cap_status_fails_on_expired_reserved_slot`).
  - Evidence: `scripts/validate-agent-docs.sh:873` reads `today = date.today()` (real clock). The validator's `__expired_reserved_slot__` check compares `today > expiry_dt`. Once today > 2026-07-20, all `reserved` slots in the fixtures fail. `test_cap_status_passes_on_valid_fixture` (currently green) will start failing on 2026-07-21. Other `assert_command_fails_with_output` fixtures may still fail, but for a different reason than the test name suggests -- the substring assertion may or may not still match depending on which error fires first.
  - Suggested fix: inject `today` into the python heredoc via env var. Bash wrapper: `CAP_STATUS_TODAY=${CAP_STATUS_TODAY:-$(date +%Y-%m-%d)}`. Python: `today = date.fromisoformat(os.environ['CAP_STATUS_TODAY'])`. Tests then set `CAP_STATUS_TODAY=2026-05-10` for a stable reference today; production runs use the default.
  - This cannot be fixed by adding more tests; it requires a small architectural change to the validator function.

- **[P3 / advisory] Validator does not cross-check `consumed` against count of authored slots.** Reviewers: maintainability + correctness (same finding, two angles). File: `scripts/validate-agent-docs.sh` (`cap_status_must_be_consistent` function). When `tier_1a_authored` is populated, the validator does not assert `consumed == len(tier_1a_authored) + count(authored_in_reserved_slots)`. After a `reserved -> authored` transition an editor could update one of `consumed`/`reserved` and forget the other; if errors are symmetric (consumed up by 1, reserved down by 1) the cap-sum still validates. `test_cap_status_fails_on_killed_missing_reason` already silently exercises this gap (consumed=4 but tier_1a_authored=[]). Suggested fix (gated_auto): add `consumed_authored = len(tier_1a_authored or []) + sum(1 for s in reserved_slots if s.get('status') == 'authored')`; emit `__consumed_authored_mismatch__:declared:counted` on disagreement. Gated because it requires a deliberate semantic decision about whether `consumed` includes Tier 1a -- plan doesn't nail this down.

- **[P3 / advisory] `REQUIRED_RESERVED_FIELDS` is stricter than Plan R1/R3 enumeration.** Reviewer: correctness. File: `scripts/validate-agent-docs.sh:817`. `REQUIRED_RESERVED_FIELDS` includes `description` and `tier`, but the plan only enumerates `slot_id`, `expiry`, `status`, `last_checked`, `required_trigger`, `trigger_source`. The live JSON happens to set all of them, so the validator currently passes; a future author writing a slot record without `description` will hit a validator failure the plan didn't warn about. Pick one direction deliberately -- either (a) accept current strictness and update the plan to enumerate `description` and `tier` as required, or (b) drop `description`/`tier` from `REQUIRED_RESERVED_FIELDS`.

- **[P3 / advisory] Bash field-parser uses `:` as delimiter for marker payloads; would mis-split on a `:` inside slot_id.** Reviewer: correctness. File: `scripts/validate-agent-docs.sh:1005-1054` (multiple case branches). `${rest%%:*}` / `${rest#*:}` would split at wrong boundary if slot_id ever contained `:`. Current `t1b-<short-name>` format is safe (hyphens, not colons), but the validator does not enforce the format. Add a slot_id format check inside the python heredoc (`if not re.match(r'^[a-z0-9._-]+$', slot_id): print(f'__bad_slot_id_format__:{slot_id}')`) with a corresponding bash case. Or migrate the marker delimiter to ASCII unit separator (`\x1f`).
