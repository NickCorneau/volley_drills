## Residual Review Findings

- **[Low — maintainability] `app/src/domain/sessionAssembly/sourceBackedReroutes.ts:?` — `destinationDrillIds` is metadata-only; no runtime read.**
  The reviewer flagged that `destinationDrillIds` is inert at runtime: the predicate only inspects `fromDrillIds`, `sessionFocus`, `playerLevel`, and `slotType`. Removing the field would shrink the surface and remove the temptation to drift it from the actual selection layer. Deliberately retained per the plan's "Key Technical Decisions" justification — `destinationDrillIds` is documentation/traceability metadata that names the source-backed sibling each reroute is courting (e.g., d49 for the d47/d48 entry), so a future maintainer can audit the registry without cross-referencing the catalog. Reconsider if the registry is ever extended past a handful of entries or if this metadata field starts going stale.

- **[Low — maintainability] `app/src/domain/sessionAssembly/sourceBackedReroutes.ts:?` — `slotType` field has zero overrides; the default `main_skill` branch is the only reachable one.**
  Every current entry implicitly defaults to `main_skill`. Reviewer suggested removing the optional `slotType` field and hard-coding the predicate's `slot.type === 'main_skill'` gate. Deliberately retained because the plan's "Deferred to Follow-Up Work" section explicitly anticipates the registry being extended to non-`main_skill` slot types when a non-main-skill source-backed sibling first appears (e.g., a warmup or finisher reroute). Removing the optional field now would force a follow-up plan to re-add the same axis. Reconsider if a year passes without a non-`main_skill` entry materializing.

- **[Low — testing] `app/src/domain/sessionAssembly/__tests__/sourceBackedReroutes.test.ts` — registry-data suite does not assert "only the d01 entry has both `sessionFocus` and `playerLevel` undefined".**
  Without this guard, a future entry that forgets to set both gates would silently match every context (effectively becoming a wildcard reroute) and the test suite would still pass. The current suite checks ID uniqueness, non-empty `fromDrillIds`, and the d01-specific empty `destinationDrillIds`, but does not encode the "default-leaf is the only ungated entry" invariant.
  Recommended next: add a vitest assertion that loops `SOURCE_BACKED_REROUTES`, asserts `entry.sessionFocus !== undefined || entry.playerLevel !== undefined` for every entry whose `id` is not `'d01-duration-fit'`, and asserts the `'d01-duration-fit'` entry has both undefined.

- **[Low — testing] `app/src/domain/sessionAssembly/__tests__/sourceBackedReroutes.test.ts` — capacity-equality boundary (capacity == planned) not directly exercised.**
  `candidateCanCarryTargetDuration` short-circuits the predicate when capacity ≥ planned, but the suite only tests strict-greater-than capacity (returns false) and strict-less-than capacity (returns true). The exact boundary `capacity === planned` is not covered.
  Recommended next: add one test case where the candidate's capacity ceiling equals the planned duration and asserts `shouldRerouteForSourceBackedSibling` returns false. Cheap to add; would harden the predicate against an off-by-one regression in `candidateCanCarryTargetDuration` semantics.
